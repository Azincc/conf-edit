# 配置对象管理器设计规格

日期：2026-07-10  
状态：待书面规格确认

## 1. 背景与目标

该项目是运行在一台 Windows 机器上的内部局域网工具。维护者通过本机控制窗口选择允许访问的 JSON 与 SQL 文件，其他同事通过浏览器按对象查看、增加、修改和删除这些文件中的配置内容。

首版必须实现：

- 管理 UTF-8 编码的 JSON 数组文件。
- 将 JSON 数组中的每个对象按 `objectName` 展示和编辑。
- 强制 JSON 顶层为数组、数组元素为对象、`objectName` 为非空字符串且在单文件内唯一。
- 管理 MySQL SQL 文件中的 `CREATE TABLE` 与对应 `INSERT INTO` 初始化语句。
- 按表名聚合 SQL 对象，展示并保留表备注、字段备注和原始 SQL 注释。
- 对 JSON 和 MySQL SQL 进行离线校验，错误内容不得写回文件。
- 只修改目标对象或目标 SQL 语句块，尽量保持其他内容、注释、空行和顺序不变。
- 保存前生成历史快照，支持差异查看和回滚。
- 检测外部修改与多人并发冲突，禁止静默覆盖。
- 最终打包为一个 Windows 可执行程序；同事端不需要安装 Python、Node 或其他运行环境。

## 2. 明确不做的内容

首版不包含：

- 登录、账号、角色或按用户授权。
- 连接真实 MySQL 实例执行或验证 SQL。
- 对 `ALTER`、`UPDATE`、`DELETE`、存储过程、触发器等语句进行结构化编辑。
- 跨 JSON 文件的 `objectName` 唯一性校验。
- 浏览器端添加任意文件路径或维护文件白名单。
- Git 提交、远程仓库同步或云端存储。
- 自动合并两个用户对同一文件的并发修改。

## 3. 技术方案

项目采用单 Python 一体化架构：

- Tkinter：Windows 控制窗口和原生文件选择器。
- Flask：REST API 与静态网页托管。
- Waitress：Windows 友好的局域网 WSGI 服务。
- 原生 HTML、CSS、JavaScript：网页交互，不使用 React 或前端构建链。
- 内置代码编辑器静态资源：提供 JSON/SQL 高亮、行号和错误定位；运行时不访问 CDN。
- Python 标准库 `json`：严格 JSON 解析。
- sqlglot：按 MySQL 方言解析和校验 SQL AST。
- SQLite：保存白名单、修订历史和应用配置。
- PyInstaller：构建单个 Windows 可执行程序。

应用为一个进程。Tkinter 运行在主线程，Waitress/Flask 运行在受控后台线程。控制窗口关闭时应先提示并安全停止服务。

## 4. 运行与控制窗口

程序启动后自动尝试启动 Web 服务，默认监听 `0.0.0.0:8765`。端口配置持久化到应用数据库；端口占用时控制窗口显示明确错误，不自动切换到不可预测的端口。

控制窗口包含：

- 服务状态：启动中、运行中、已停止或启动失败。
- 本机访问地址与可用局域网地址。
- “在浏览器中打开”“复制地址”“启动”“停止”“重启”操作。
- 当前允许访问文件列表，展示显示名、类型、路径和有效状态。
- “添加 JSON”“添加 SQL”“移除”操作。
- 通过 Windows 原生文件选择器选择文件。
- 最近运行日志与错误摘要。

添加白名单文件时：

1. 将路径解析为规范化绝对路径。
2. 确认目标是普通文件且扩展名为 `.json` 或 `.sql`。
3. 确认文件能按 UTF-8 解码。
4. 执行对应的完整语法检查并记录状态；语法损坏的文件仍可加入，以便通过整文件修复模式处理。
5. 防止同一路径被重复添加。
6. 将显示名、规范路径和文件类型保存到 SQLite。

浏览器 API 永远只接收白名单文件 ID，不接受文件路径。白名单修改在服务运行期间立即生效。

## 5. 网页信息架构

网页采用已确认的“对象列表 + 编辑抽屉”布局：

- 顶部：产品名称、当前文件状态、刷新与历史入口。
- 左侧：按 JSON/SQL 分组的白名单文件列表，显示有效、错误、丢失或不可写状态。
- 中央：当前文件的对象列表、搜索、对象数量、新增按钮和校验状态。
- 右侧抽屉：新增或编辑当前对象。
- 窄屏时左侧栏可折叠，右侧抽屉改为全屏层；核心能力仍可使用。

JSON 对象列表至少显示 `objectName`、对象字段数和校验状态。SQL 对象列表至少显示表名、字段数、初始化语句数和备注摘要。

所有删除与回滚操作都需要二次确认。保存按钮在编辑内容未通过校验时保持禁用，并同时显示可定位的错误信息。

## 6. JSON 对象模型与编辑

### 6.1 文件规则

- 顶层值必须是数组。
- 数组每个元素必须是 JSON Object，不接受字符串、数字、数组、布尔值或 null。
- 每个对象必须包含 `objectName`。
- `objectName` 必须是去除首尾空白后仍非空的字符串。
- `objectName` 在当前文件内唯一，大小写敏感。
- 严格遵循 JSON，不接受注释、尾随逗号、单引号或未加引号的属性名。

### 6.2 保留源文件格式

后端使用基于 `json.JSONDecoder.raw_decode` 的数组元素扫描器，在完成严格解析的同时记录每个顶层元素的字符范围。编辑时只替换目标元素的范围：

- 更新：替换目标对象文本，保留其他元素原文。
- 删除：移除目标对象及正确一侧的逗号和相邻结构空白。
- 新增：在数组结束符前插入新对象，沿用文件的换行符和缩进风格。

提交的对象文本在写入前单独解析并再次放入拟写回的完整文件中校验。字段顺序按用户提交内容保留。编辑器提供格式化按钮，但保存不强制重排未修改对象。

修改 `objectName` 视为重命名；保存时使用原名称定位对象，并校验新名称没有与同文件其他对象重复。

### 6.3 损坏文件

若现有文件不能解析，对象列表替换为文件错误页，展示错误行、列、上下文和原因。用户可以打开“整文件修复”编辑器，但只有修复后的完整内容通过全部 JSON 规则后才能保存。

## 7. SQL 对象模型与编辑

### 7.1 语句切分

后端使用保留源码位置的 SQL 词法切分器，在字符串、反引号标识符、行注释和块注释之外识别分号。每个原始语句块保留其字符范围和前导注释，再交给 sqlglot 按 `mysql` 方言解析。

`DELIMITER`、存储过程和包含自定义分隔符的脚本不在首版支持范围；遇到时作为明确的文件校验错误报告。

### 7.2 SQL 对象聚合

- 每个 `CREATE TABLE` 产生一个表对象。
- 所有目标表相同的 `INSERT INTO` 归入该表对象。
- 表名匹配忽略大小写，展示和写回时保留源码中的原始拼写及反引号。
- 同一文件内不允许出现两个同名 `CREATE TABLE`。
- 没有对应 `CREATE TABLE` 的 `INSERT INTO` 作为文件错误报告。
- 能正确解析但不属于 `CREATE TABLE` 或 `INSERT INTO` 的语句显示为只读“未管理语句”，原样保留。
- 无法解析的任何语句都会使文件进入错误状态，保存前必须修复。

### 7.3 SQL 编辑抽屉

SQL 对象编辑抽屉包含：

- 建表语句编辑器：必须且只能包含目标表的一条 `CREATE TABLE`。
- 初始化语句编辑器：可包含零条或多条只针对目标表的 `INSERT INTO`。
- 备注摘要：从列定义、表选项和原始注释中提取表备注、字段备注。
- 校验结果：语法错误、表名不一致、重复表名、INSERT 字段和值数量不一致等。

更新时只替换目标 `CREATE TABLE` 范围，并移除该表原有的所有 `INSERT INTO` 范围；新的初始化语句紧跟在建表语句之后写入。其他表和未管理语句保持原样。

修改建表语句中的表名视为重命名。所有初始化语句必须使用新表名，且新表名不得与当前文件其他表冲突。

新增表对象追加在文件末尾，并沿用现有换行风格。删除表对象会删除该表的建表语句和全部初始化语句，但不删除无法确定归属的未管理语句。

### 7.4 离线 SQL 校验

首版校验包括：

- 每条语句可由 MySQL 方言解析。
- 建表对象恰好包含一条 `CREATE TABLE`。
- 建表名和所有初始化语句目标表一致。
- 单文件表名不重复。
- INSERT 显式字段列表不存在重复字段。
- 每一行 VALUES 数量与 INSERT 字段数量一致。
- 未提供字段列表时，VALUES 数量与建表字段数量一致。
- INSERT 引用的显式字段存在于建表字段中。

不校验数据库中已有对象、外键目标是否真实存在、权限、字符集是否安装或语句实际执行结果。

## 8. 保存、冲突与历史

### 8.1 修订标识

读取文件时计算 SHA-256，并作为 `revision` 返回浏览器。所有新增、修改、删除、整文件修复和回滚请求必须携带加载时的 `revision`。

### 8.2 写入流程

每个白名单文件在进程内拥有独立锁。由于文件系统和 SQLite 不能共享一个原子事务，写入使用可恢复的两阶段修订日志：

1. 获取文件锁。
2. 重新读取文件并计算当前 SHA-256。
3. 当前哈希与请求 `revision` 不同则返回 HTTP 409，不执行覆盖。
4. 基于当前源码范围生成候选完整文件。
5. 对候选完整文件执行对应的全部校验。
6. 在 SQLite 中提交一条 `PENDING` 修订，保存修改前后完整内容、哈希、动作和来源信息。
7. 在目标文件同目录写入临时文件，刷新内容并使用 `os.replace` 原子替换。
8. 将修订状态更新为 `APPLIED`。
9. 返回新 `revision` 和对象索引。

若文件系统替换失败，将修订标记为 `FAILED`，原文件保持不变。若进程在文件替换后、状态更新前退出，下一次启动通过磁盘哈希与修订的前后哈希比对完成恢复：匹配新哈希则标记为 `APPLIED`，匹配旧哈希则标记为 `FAILED`，两者都不匹配则标记为冲突并进入只读保护。若历史数据库不可写，禁止修改源码文件。

### 8.3 外部修改冲突

HTTP 409 响应包含当前哈希、用户基准版本与磁盘当前版本的文本差异摘要。界面保留用户编辑内容，并提供：

- 复制当前编辑内容。
- 查看差异。
- 重新加载磁盘版本。

首版不提供“强制覆盖”或自动三方合并。

### 8.4 历史数据

SQLite 为每次成功写入保存：

- 文件 ID。
- 递增版本号。
- 修改前和修改后的完整文件内容，使用 zlib 压缩。
- 修改前和修改后的 SHA-256。
- 修订状态：`PENDING`、`APPLIED`、`FAILED` 或 `CONFLICTED`。
- 动作类型：新增、修改、删除、修复或回滚。
- 目标对象名称或表名。
- 客户端 IP。
- 可选修改说明。
- 创建时间。

文件首次被修改前先建立基线版本。历史页默认只将 `APPLIED` 修订计入可回滚版本，可查看失败或冲突修订的诊断信息。历史页可比较任意相邻版本或选定版本与当前版本。回滚会将选定版本内容作为一次新的写入，因此历史链不会被截断。

## 9. API 边界

核心接口按以下职责划分：

- `GET /api/status`：服务状态与版本。
- `GET /api/files`：白名单文件及校验状态。
- `GET /api/files/{file_id}/objects`：对象索引和当前修订。
- `GET /api/files/{file_id}/object?key=...`：读取单个对象或表对象。
- `POST /api/files/{file_id}/validate`：校验编辑器内容，不写文件。
- `POST /api/files/{file_id}/objects`：新增对象。
- `PUT /api/files/{file_id}/object`：修改或重命名对象。
- `DELETE /api/files/{file_id}/object`：删除对象。
- `PUT /api/files/{file_id}/repair`：修复整个损坏文件。
- `GET /api/files/{file_id}/history`：历史列表。
- `GET /api/files/{file_id}/history/{version}/diff`：历史差异。
- `POST /api/files/{file_id}/history/{version}/rollback`：回滚。

API 错误统一返回稳定错误码、用户可读消息和可选的行号、列号、语句索引、字段名或差异信息。

## 10. 局域网与文件安全

系统没有账号验证，因此任何能够访问服务地址的局域网用户都拥有同等编辑权限。控制窗口必须明确显示这一点。

仍需实现以下边界：

- API 只通过数据库中的白名单文件 ID 定位路径。
- 每次访问重新确认规范路径仍指向同一普通文件。
- 禁止目录、未知扩展名和非 UTF-8 内容。
- 静态页面与 API 同源，不开放 CORS。
- 每次启动生成请求防伪令牌，修改请求必须携带同源页面获得的令牌。
- 校验 Host/Origin，降低跨站请求和 DNS 重绑定风险。
- 响应不返回完整服务端文件路径；网页只显示配置的文件显示名。
- 日志避免记录完整文件内容和防伪令牌。

## 11. 错误处理

- 文件不存在或无读取权限：文件列表显示不可用，不导致服务崩溃。
- 文件无写入权限：允许只读查看，禁用修改操作。
- JSON/SQL 校验失败：返回精确诊断，绝不写盘。
- 文件锁冲突：短暂等待后返回“文件正被修改”，用户可重试。
- 外部修改：返回 409 并要求重新加载。
- SQLite 不可用：进入只读保护状态。
- 原子替换失败：清理临时文件，保留原文件，并将待处理修订标记为失败。
- 启动时发现未完成修订：按前后哈希自动恢复状态；无法判断时将文件置为只读冲突状态。
- 未捕获异常：返回通用错误 ID，将堆栈写入本机日志，不泄露给浏览器。
- 服务端口占用：控制窗口显示端口和处理建议。

## 12. 数据位置与打包

应用数据库和日志放在当前 Windows 用户的本地应用数据目录，例如：

`%LOCALAPPDATA%\ConfEdit\conf-edit.db`  
`%LOCALAPPDATA%\ConfEdit\logs\app.log`

源码文件继续位于用户选择的原始代码仓库中。PyInstaller 打包产物包含 Python 解释器、依赖、网页资源和编辑器静态资源，不要求目标机器安装开发环境。

## 13. 测试策略

使用 pytest，并遵循先写失败测试再实现的流程。

### 13.1 单元测试

- JSON 数组扫描、嵌套结构、转义字符、Unicode、不同换行和缩进。
- JSON 新增、更新、删除、重命名和单文件重复名称。
- JSON 局部替换后未修改对象逐字节保持不变。
- SQL 分号、字符串、反引号、行注释和块注释切分。
- MySQL 建表与 INSERT 聚合、备注提取、表重命名和未管理语句。
- INSERT 字段和值数量检查。
- 文本差异和修订哈希。

### 13.2 服务与存储测试

- 临时目录中的白名单规范化和越权路径拒绝。
- SQLite 基线、连续修订、差异和回滚。
- 文件锁、外部修改 409、原子替换失败和历史失败保护。
- Flask API 成功与错误响应契约。
- CSRF、Origin、Host 和 CORS 边界。

### 13.3 端到端与打包测试

- 使用 Python Playwright 启动真实服务，覆盖选择文件、JSON CRUD、SQL CRUD、校验错误、冲突和回滚。
- 控制窗口业务逻辑与 Tkinter 视图分离，业务逻辑可无界面测试；Windows 上执行一次真实 GUI 冒烟测试。
- 构建 PyInstaller 可执行文件，在干净 Windows 环境中验证启动、文件选择、局域网访问和关闭。

## 14. 完成标准

满足以下条件即认为首版完成：

- 维护者可以通过 Windows 控制窗口维护 JSON/SQL 文件白名单并稳定启停服务。
- 局域网同事可以按文件、按对象浏览和搜索。
- JSON 对象和 SQL 表对象均可新增、修改、删除。
- 所有错误 JSON 和不符合支持范围的错误 SQL 都无法保存。
- 修改只影响目标对象或目标语句块，测试证明未修改内容保持原样。
- 多人或外部编辑产生的文件哈希冲突不会被覆盖。
- 每次成功写入都有可查看、可比较、可回滚的历史。
- 单个 Windows 可执行程序可在未安装 Python/Node 的机器上运行。
